## 연구 과제 1: 이력서 마크다운 테이블 파싱 전략

### 결정: 커스텀 텍스트 파서 (Vanilla JavaScript)

### 근거

실제 이력서 파일(`data/resume/Hangwoo_Intern_Resume.docx.md`)을 분석한 결과,
이 파일은 일반적인 마크다운(heading + paragraph)이 아니라 **마크다운 테이블 형식**이다.

- 전체 내용이 `| col1 | col2 | col3 | col4 |` 형태의 테이블 행으로 구성
- 섹션 구분자: `**EDUCATION**`, `**WORK EXPERIENCE**`, `**SKILLS AND INFORMATION**` 등의
  볼드 텍스트가 테이블 셀 안에 존재
- 경력 항목: 회사명(볼드) + 직책(이탤릭) + 날짜(볼드)가 파이프(`|`)로 구분된 셀에 분산
- 상세 내용: 다음 행의 첫 번째 셀에 장문의 텍스트로 포함

이러한 구조는 `marked.js` 같은 범용 마크다운 라이브러리로 렌더링해도
단순한 HTML 테이블만 생성되므로, **목적에 맞지 않는다**.

### 대안 검토

| 대안 | 장점 | 채택하지 않은 이유 |
|------|------|---------------------|
| marked.js CDN | 범용 마크다운 지원 | 테이블을 그대로 렌더링하므로 "구조적 재구성" 요구사항(FR-004) 위반 |
| showdown.js CDN | marked.js와 유사 | 같은 이유 |
| 커스텀 파서 (런타임) | JS에서 fetch로 md 파일 읽기 | file:// 프로토콜에서 fetch가 CORS로 차단됨 |
| **커스텀 파서 (빌드 타임)** | Node.js로 md 파싱 → HTML 직접 생성 | ✅ 채택 — file:// 제약 없음, CDN 불필요, 가장 단순 |

### 파싱 전략 상세

1. `generate.js`에서 이력서 마크다운을 **줄 단위로** 읽는다
2. 테이블 행(`|`로 시작)을 파싱하여 셀 내용을 추출한다
3. 섹션 키워드(`EDUCATION`, `WORK EXPERIENCE`, `EXTRA-CURRICULAR ACTIVITIES`, `SKILLS AND INFORMATION`)를 감지하여 섹션별 데이터를 분리한다
4. 볼드(`**...**`), 이탤릭(`*...*`) 마크다운 서식을 정규식으로 제거/변환한다
5. 파싱된 구조화 데이터를 HTML 템플릿에 삽입하여 포트폴리오 UI를 생성한다

## 연구 과제 2: file:// 프로토콜 제약과 생성 방식

### 결정: Node.js 생성 스크립트 (`generate.js`)

### 근거

`output/index.html`을 `file://` 프로토콜로 열 때의 제약:
- `fetch()`, `XMLHttpRequest`는 **CORS 정책으로 차단**됨 (file:// → file:// 요청 불가)
- 외부 파일(이미지, CSS, JS) 참조는 **상대 경로**로만 가능
- `<script type="module">`은 일부 브라우저에서 file://과 호환 불가

따라서 **런타임에 data/ 폴더를 읽는 방식은 불가능**하다.
대신 **빌드 타임(생성 시점)**에 모든 데이터를 처리하여 완성된 HTML을 생성해야 한다.

### Node.js 생성 스크립트 접근 방식

- `node generate.js` 한 줄로 실행
- npm install 불필요 (Node.js 내장 `fs`, `path` 모듈만 사용)
- Constitution의 "빌드 과정 금지"에 대한 해석:
  - Webpack, Vite 등의 **번들러/빌드 도구**를 금지하는 것이지,
    단순한 파일 생성 스크립트까지 금지하는 것은 아님
  - `generate.js`는 data/를 읽어 output/를 생성하는 **일회성 변환 스크립트**
  - 의존성 설치, 설정 파일, 워치 모드 등이 없으므로 "빌드 과정"에 해당하지 않음

### 대안 검토

| 대안 | 장점 | 채택하지 않은 이유 |
|------|------|---------------------|
| 모든 것을 수동 작성 | 빌드 과정 완전 제거 | "자동 생성" 요구사항에 부합하지 않음 |
| Python 스크립트 | Python 익숙한 사용자에게 편리 | Node.js가 더 보편적, 웹 프로젝트와 자연스러운 조합 |
| Shell 스크립트 | 매우 단순 | 마크다운 파싱이 어려움, 크로스 플랫폼 호환성 낮음 |
| **Node.js 스크립트** | fs/path 내장, 정규식 편리, 설치 불필요 | ✅ 채택 |

## 연구 과제 3: 이미지 처리 방식

### 결정: data/photos/ → output/photos/ 파일 복사 + 상대 경로 참조

### 근거

`file://` 프로토콜에서 이미지를 표시하려면:
- 이미지 파일이 HTML 파일과 **같은 디렉토리 트리** 안에 있어야 함
- `../../data/photos/` 같은 상위 경로 참조는 가능하나, output/ 폴더를 독립적으로
  이동·배포할 수 없어 비직관적
- `output/photos/`에 복사하고 `photos/파일명`으로 참조하는 것이 가장 단순하고 직관적

### 대안 검토

| 대안 | 장점 | 채택하지 않은 이유 |
|------|------|---------------------|
| Base64 인라인 이미지 | 단일 HTML 파일 | 파일 크기 폭증, 4장 JPEG 시 수 MB |
| 상위 경로 참조 (`../../data/photos/`) | 복사 불필요 | output/ 폴더 이동 시 깨짐, 비직관적 |
| **output/photos/ 복사** | 직관적, 독립적, 표준적 | ✅ 채택 |

## 연구 과제 4: CSS/JS 인라인 vs 외부 파일

### 결정: CSS와 JS를 index.html에 인라인으로 포함

### 근거

- Constitution 원칙 I(단순함): 파일 개수 최소화
- 사용자는 `output/index.html` 하나만 신경 쓰면 됨 (+ photos/ 폴더)
- CSS와 JS 양이 소규모(각각 100~200줄 이하 예상)이므로 인라인이 적합
- 별도 파일 분리는 오히려 관리 포인트를 늘려 Constitution 위반에 가까움

### 대안 검토

| 대안 | 채택하지 않은 이유 |
|------|---------------------|
| style.css + script.js 외부 파일 | 파일 3개로 증가, 단순함 원칙 위반 |
| **인라인 (index.html 내 `<style>` + `<script>`)** | ✅ 채택 — 최소 파일 수 |

## 연구 과제 5: 이력서 섹션별 파싱 규칙

### 결정: 섹션 키워드 기반 상태 머신 파서

### 근거

실제 이력서 구조 분석 결과:

```
행 1: | Hangwoo Cho |  |  |  |            → 이름 추출
행 2: | +82... | email |  |  |             → 연락처 추출
행 3: 구분선 (| ----- | :---- | ...)       → 무시
행 4: | **EDUCATION** |  |  |  |           → 섹션 전환: education
행 5: | **학위 - 학교** |  | **기간** |     → education 항목
행 6: | GPA 정보 |  |  |  |               → education 보조 정보
행 7: | **WORK EXPERIENCE** |  |  |  |     → 섹션 전환: work
행 8: | **회사명** *직책* |  | **장소** *기간* | → work 항목 헤더
행 9: | 상세 설명... |  |  |  |            → work 항목 설명
...
행 N: | **SKILLS AND INFORMATION** |       → 섹션 전환: skills
```

파싱 알고리즘:
1. 현재 섹션 상태를 추적하는 변수 (`currentSection`)
2. 각 행을 `|`로 분리하여 셀 추출
3. 볼드 키워드(`**EDUCATION**` 등)로 섹션 전환 감지
4. 섹션별로 다른 파싱 규칙 적용:
   - **header**: 이름(첫 행 첫 셀), 연락처(둘째 행)
   - **education**: 학위+학교(1열), 기간(3열), GPA(다음 행)
   - **work**: 회사+직책(1열), 장소+기간(3열), 설명(다음 행)
   - **activities**: work와 동일 구조
   - **skills**: key-value 쌍 (1열: 키, 2열: 값)
