# Feature Specification: 개발자 포트폴리오 웹사이트 자동 생성기

**Feature Branch**: `002-portfolio-site-gen`  
**Created**: 2026-03-18  
**Status**: Draft  
**Input**: 사용자 설명 — "개발자 개인을 소개하는 단일 페이지 포트폴리오 웹사이트를 자동으로 생성하는 도구"

## User Scenarios & Testing *(mandatory)*

### User Story 1 — 개발자 첫인상 전달 (Priority: P1)

방문자(채용 담당자, 동료 개발자 등)가 페이지에 접속하면, 개발자의 이름·한 줄 소개·프로필 사진이 가장 먼저 보여야 한다. 방문자는 5초 이내에 "이 사람이 누구인지"를 파악할 수 있어야 한다.

**Why this priority**: 포트폴리오 웹사이트의 핵심 목적은 개발자를 소개하는 것이다. 첫인상 영역 없이는 나머지 콘텐츠가 의미를 갖지 못한다.

**Independent Test**: `output/index.html`을 브라우저에서 열었을 때, 스크롤 없이 이름·한 줄 소개·프로필 사진이 화면에 보이는지 확인한다.

**Acceptance Scenarios**:

1. **Given** `data/resume/` 폴더에 이력서 마크다운 파일이 존재하고, `data/photos/` 폴더에 이미지가 1장 이상 존재할 때, **When** `output/index.html`을 브라우저에서 열면, **Then** 페이지 최상단에 개발자 이름, 한 줄 소개, 대표 사진이 표시된다.
2. **Given** 방문자가 페이지에 처음 접속했을 때, **When** 화면을 보면, **Then** 스크롤 없이 개발자의 정체성을 5초 이내에 파악할 수 있다.
3. **Given** 페이지가 로드되었을 때, **When** 소개 영역의 디자인을 확인하면, **Then** 미니멀하고 모던한 스타일이 적용되어 있고, 기본 HTML 느낌이 나지 않는다.

---

### User Story 2 — 이력서 구조적 표시 (Priority: P2)

방문자가 개발자의 경력, 학력, 기술 스택 등을 포트폴리오 형태로 정돈되어 읽기 쉽게 확인할 수 있어야 한다. 이력서 마크다운 파일의 내용을 단순히 렌더링하는 것이 아니라, 정보를 구조적으로 해석하여 포트폴리오에 적합한 형태로 재구성해야 한다.

**Why this priority**: 채용 담당자가 가장 관심 있는 정보는 경력과 역량이다. 이력서 정보를 구조적으로 표시하는 것은 포트폴리오의 핵심 가치이다.

**Independent Test**: `data/resume/` 폴더에 마크다운 이력서를 넣고 `output/index.html`을 열었을 때, Education, Work Experience, Skills 등이 포트폴리오 스타일의 별도 섹션으로 분리·표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** `data/resume/` 폴더에 마크다운 이력서가 존재할 때, **When** 페이지를 열면, **Then** 학력(Education), 경력(Work Experience), 기술(Skills) 등이 시각적으로 구분된 섹션으로 표시된다.
2. **Given** 이력서에 경력 사항이 포함되어 있을 때, **When** 경력 섹션을 보면, **Then** 회사명, 직책, 기간, 주요 성과가 정리된 형태로 표시된다.
3. **Given** 이력서 마크다운 원본이 있을 때, **When** 페이지에서 이력 정보를 확인하면, **Then** 마크다운이 그대로 출력된 것이 아니라 포트폴리오에 적합한 UI 형태로 재구성되어 있다.

---

### User Story 3 — 사진 갤러리 (Priority: P3)

방문자가 개발자의 사진 또는 프로젝트 이미지를 시각적으로 편하게 볼 수 있어야 한다. `data/photos/` 폴더의 이미지를 자동으로 탐색하여 깔끔한 그리드 갤러리로 표시한다.

**Why this priority**: 시각적 콘텐츠는 페이지의 인상을 크게 좌우하지만, 소개와 이력서 없이도 독립적으로 동작할 수 있다.

**Independent Test**: `data/photos/` 폴더에 이미지를 넣고 `output/index.html`을 열어 갤러리 섹션에 이미지가 그리드 형태로 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** `data/photos/` 폴더에 이미지 파일(JPEG, PNG 등)이 존재할 때, **When** 페이지를 열면, **Then** 이미지들이 2~3열 그리드 레이아웃의 갤러리로 표시된다.
2. **Given** 갤러리에 이미지가 표시되어 있을 때, **When** 이미지 위에 마우스를 올리면, **Then** 살짝 확대 또는 그림자 효과가 적용된다.
3. **Given** `data/photos/` 폴더에 이미지가 추가되었을 때, **When** 페이지를 다시 생성하면, **Then** 추가된 이미지가 갤러리에 자동으로 포함된다.

---

### User Story 4 — 직관적 UI와 세련된 디자인 (Priority: P4)

방문자가 복잡하지 않고 직관적인 UI를 통해 신뢰감을 느낄 수 있어야 한다. 전체 페이지의 디자인이 미니멀하고 모던하며, 기본 HTML 느낌이 나지 않아야 한다.

**Why this priority**: 디자인 품질은 모든 콘텐츠 위에 적용되는 것으로, 다른 스토리와 함께 점진적으로 완성된다. 독립적으로도 테스트 가능하지만, 콘텐츠가 있을 때 더 의미가 있다.

**Independent Test**: 페이지를 열고 중앙 정렬 컨테이너, 충분한 여백, 일관된 타이포그래피, 색상 체계가 적용되어 있는지 시각적으로 점검한다.

**Acceptance Scenarios**:

1. **Given** 페이지가 생성되었을 때, **When** 브라우저에서 열면, **Then** 중앙 정렬 컨테이너(최대 너비 800~1000px)가 적용되어 있다.
2. **Given** 페이지의 각 섹션을 확인할 때, **When** 여백을 측정하면, **Then** 섹션 간 40px 이상의 여백이 확보되어 있다.
3. **Given** 페이지의 텍스트를 확인할 때, **When** 타이포그래피를 보면, **Then** sans-serif 폰트가 사용되고, 줄 간격이 1.5 이상이며, 제목과 본문 크기가 명확히 구분된다.
4. **Given** 페이지의 색상을 확인할 때, **When** 배경·텍스트·포인트 컬러를 보면, **Then** 흰색/연한 회색 배경, 짙은 회색 텍스트, 단일 포인트 컬러가 적용되어 대비가 명확하다.

---

### User Story 5 — 로컬 파일 직접 실행 (Priority: P5)

사용자(포트폴리오 소유자)가 `output/index.html` 파일을 더블클릭하는 것만으로 브라우저에서 웹페이지를 바로 확인할 수 있어야 한다. 별도의 서버 실행, 빌드 과정, 명령어 입력이 필요하지 않아야 한다.

**Why this priority**: 이것은 "파일을 넣으면 결과가 나오고, 클릭하면 바로 열리는" 핵심 경험의 최종 단계이다. 다른 모든 스토리의 결과물이 이 경험으로 수렴한다.

**Independent Test**: `output/index.html` 파일을 Finder에서 더블클릭하여 브라우저에서 정상 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** `output/` 폴더에 생성된 `index.html` 파일이 있을 때, **When** 파일을 더블클릭하면, **Then** 기본 브라우저에서 포트폴리오 페이지가 정상적으로 열린다.
2. **Given** 별도의 서버가 실행되지 않은 상태에서, **When** `output/index.html`을 열면, **Then** 모든 이미지, 스타일, 콘텐츠가 깨지지 않고 정상적으로 표시된다.
3. **Given** `data/` 폴더의 내용을 수정(사진 추가, 이력서 변경)한 후, **When** 페이지를 다시 생성하고 `output/index.html`을 열면, **Then** 변경된 내용이 반영되어 있다.

---

### Edge Cases

- `data/photos/` 폴더가 비어 있는 경우: 갤러리 섹션을 숨기거나, "사진이 없습니다"와 같은 안내 메시지를 표시한다.
- `data/resume/` 폴더에 마크다운 파일이 없는 경우: 이력서 섹션을 숨기거나, 기본 안내 메시지를 표시한다.
- 이미지 파일이 매우 큰 경우(>5MB): 페이지 로딩이 느려질 수 있으나, 별도의 이미지 최적화 처리는 이 스펙의 범위 밖으로 한다.
- 이력서 마크다운의 구조가 예상과 다른 경우(테이블 형식, 비표준 형식 등): 최선의 노력으로 주요 섹션(Education, Work Experience, Skills)을 파싱하되, 파싱에 실패한 부분은 원본 마크다운을 정리된 형태로 표시한다.
- `data/photos/` 폴더에 이미지가 아닌 파일이 섞여 있는 경우: 이미지 확장자(jpg, jpeg, png, gif, webp)만 필터링하여 표시한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 `data/resume/` 폴더의 마크다운 파일을 읽어 개발자의 이름과 한 줄 소개를 추출하여 페이지 최상단 소개 영역에 표시해야 한다.
- **FR-002**: 시스템은 `data/photos/` 폴더의 첫 번째 이미지를 대표 프로필 사진으로 소개 영역에 표시해야 한다.
- **FR-003**: 시스템은 이력서 마크다운의 내용을 구조적으로 해석하여, 학력·경력·기술 등을 포트폴리오 형태의 별도 섹션으로 재구성하여 표시해야 한다.
- **FR-004**: 시스템은 이력서 마크다운의 원본을 그대로 출력해서는 안 된다. 반드시 정보를 파싱하고 포트폴리오 UI로 재구성해야 한다.
- **FR-005**: 시스템은 `data/photos/` 폴더의 이미지 파일(jpg, jpeg, png, gif, webp)을 자동으로 탐색하여 그리드 갤러리로 표시해야 한다.
- **FR-006**: 갤러리는 2~3열 그리드 레이아웃으로 구성하고, 이미지에 border-radius를 적용하며, hover 시 확대 또는 그림자 효과를 제공해야 한다.
- **FR-007**: 전체 페이지는 중앙 정렬 컨테이너(최대 너비 800~1000px)를 적용하고, 섹션 간 40px 이상의 여백을 확보해야 한다.
- **FR-008**: 타이포그래피는 sans-serif 폰트(system-ui 등)를 사용하고, 줄 간격 1.5 이상을 확보하며, 제목과 본문 크기를 명확히 구분해야 한다.
- **FR-009**: 색상은 흰색 또는 연한 회색(#f9f9f9) 배경, 짙은 회색(#222) 텍스트, 단일 포인트 컬러를 사용하여 대비가 명확해야 한다.
- **FR-010**: 최종 결과물은 `output/` 폴더에 생성되어야 하며, `output/index.html`을 더블클릭하여 별도의 서버 없이 로컬에서 바로 동작해야 한다.
- **FR-011**: 시스템은 HTML, CSS, Vanilla JavaScript만 사용해야 한다. 프레임워크, 번들러, 빌드 도구, 패키지 매니저를 사용해서는 안 된다.
- **FR-012**: 마크다운 렌더링 등 필수적인 경우 CDN 기반 경량 라이브러리 사용을 허용하되, 사용 사유를 명시해야 한다.
- **FR-013**: `data/photos/` 폴더가 비어 있는 경우, 갤러리 섹션을 숨기거나 안내 메시지를 표시해야 한다.
- **FR-014**: `data/resume/` 폴더에 마크다운 파일이 없는 경우, 이력서 섹션을 숨기거나 안내 메시지를 표시해야 한다.

### Key Entities

- **개발자 프로필**: 이름, 한 줄 소개, 대표 사진. 이력서 마크다운에서 추출한 핵심 정보.
- **이력서 데이터**: 학력(Education), 경력(Work Experience), 기술(Skills) 등 이력서 마크다운을 파싱하여 추출한 구조화된 정보 모음.
- **사진 컬렉션**: `data/photos/` 폴더에서 자동 탐색된 이미지 파일 목록. 갤러리 표시에 사용.
- **포트폴리오 페이지**: 위 엔티티들을 조합하여 최종 생성된 단일 HTML 페이지(`output/index.html`).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: `output/index.html`을 더블클릭하면 브라우저에서 모든 콘텐츠(소개, 이력서, 갤러리)가 정상적으로 표시된다.
- **SC-002**: 방문자가 페이지 접속 후 5초 이내에 개발자의 이름, 역할, 핵심 역량을 파악할 수 있다.
- **SC-003**: `data/` 폴더의 내용(사진 추가, 이력서 수정)을 변경한 후 페이지를 다시 생성하면 변경 사항이 결과에 반영된다.
- **SC-004**: 페이지 디자인이 미니멀하고 모던하여, 기본 HTML 느낌이 아닌 "세련된 포트폴리오"로 인식된다.
- **SC-005**: 이력서 정보가 마크다운 원본 그대로가 아닌, 포트폴리오 형태로 재구성되어 표시된다.
- **SC-006**: 별도의 서버 실행, 빌드 과정, 명령어 입력 없이 로컬 파일 시스템에서 바로 동작한다.

## Assumptions

- 이력서 마크다운 파일은 영어로 작성 되어 있다고 가정한다 (현재 `data/resume/Hangwoo_Intern_Resume.docx.md` 기준).
- 이력서 마크다운은 테이블 기반 구조로, Education / Work Experience / Skills 등의 주요 섹션 키워드를 포함하고 있다고 가정한다.
- `data/photos/` 폴더에는 일반적인 웹 이미지 형식(JPEG, PNG 등)이 포함되어 있다고 가정한다.
- 대표 프로필 사진은 `data/photos/` 폴더의 첫 번째 이미지를 사용하며, 별도의 지정 방식은 이 스펙의 범위 밖으로 한다.
- 성능 목표: 일반적인 웹 브라우저(Chrome, Safari, Firefox)에서 3초 이내에 페이지가 완전히 로드된다고 가정한다.
- 마크다운 렌더링을 위해 CDN 기반 경량 라이브러리(예: marked.js)를 사용할 수 있다.
