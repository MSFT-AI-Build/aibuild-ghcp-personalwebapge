# 정적 웹 포트폴리오 생성기 Constitution

## 핵심 원칙

### I. 단순함이 최우선이다 (비타협)

- 모든 설계와 구현 결정에서 단순함이 다른 모든 가치보다 우선한다.
- 파일 개수는 반드시 최소화해야 한다.
- 단일 페이지(single-page) 구조만 허용한다.
- 불필요한 추상화, 헬퍼 함수, 유틸리티 모듈 생성을 금지한다.
- "나중에 필요할 수도 있다"는 이유로 코드를 추가해서는 안 된다.
- 근거: 이 프로젝트의 핵심 가치는 "파일을 넣으면 포트폴리오가
  생성되고, 클릭하면 바로 열리는" 경험이다. 복잡성은 이
  경험을 파괴한다.

### II. 비개발자가 이해할 수 있어야 한다

- 코드와 프로젝트 구조는 개발 경험이 없는 사람도
  흐름을 파악할 수 있을 정도로 명확해야 한다.
- 변수명, 파일명, 폴더명은 반드시 직관적이고
  서술적이어야 한다.
- 전문 용어 사용을 최소화하고, 사용할 경우 반드시
  주석으로 설명을 첨부해야 한다.
- 근거: 이 프로젝트의 대상 사용자는 비개발자를
  포함한다. 이해할 수 없는 결과물은 실패한 것이다.

### III. 구조가 직관적이어야 한다

- 폴더 구조만 보고도 프로젝트의 동작 방식을 유추할
  수 있어야 한다.
- 입력 데이터 경로: `data/photos` (사진·이미지),
  `data/resume` (이력서 마크다운 파일).
- 출력 경로: `output/` 폴더에 최종 결과물을 생성한다.
- 사용자는 `output/index.html`을 클릭하여 별도의
  서버 실행 없이 로컬에서 바로 웹페이지를 열 수
  있어야 한다.
- 근거: 코드보다 구조가 먼저 이해되어야 한다.
  구조가 곧 문서이다.

### IV. 읽기 쉬움을 우선한다

- 유지보수성, 확장성, 재사용성보다 "읽기 쉬움"을
  우선한다.
- 코드는 위에서 아래로 순서대로 읽을 수 있도록
  작성해야 한다.
- 복잡한 제어 흐름(깊은 중첩, 콜백 체인 등)을
  금지한다.
- 주석은 "왜(why)"를 설명할 때만 사용하고, "무엇(what)"은
  코드 자체가 설명해야 한다.
- 근거: 한 번 읽고 바로 이해할 수 없는 코드는 이
  프로젝트의 목적에 부합하지 않는다.

## 기술 제약

- **허용 기술**: HTML, CSS, Vanilla JavaScript만
  사용한다.
- **백엔드 금지**: 서버 사이드 코드, 데이터베이스,
  API 서버를 사용해서는 안 된다.
- **프레임워크 금지**: React, Vue, Angular, Next.js,
  Svelte 등 모든 프론트엔드 프레임워크를
  사용해서는 안 된다.
- **빌드 과정 금지**: Webpack, Vite, Rollup 등
  번들러나 빌드 도구를 사용해서는 안 된다.
- **패키지 매니저 금지**: npm, yarn, pnpm 등
  패키지 매니저를 사용해서는 안 된다.
- **CDN 예외**: 마크다운 렌더링 등 필수적인 경우에
  한해 CDN 기반 경량 라이브러리만 허용한다.
  반드시 사용 사유를 명시해야 한다.

## 입출력 워크플로우

- **입력**:
  - `data/photos/` — 개발자 사진 또는 프로젝트 이미지
  - `data/resume/` — 개발자 이력서 마크다운 파일
- **출력**:
  - `output/` — 최종 생성된 포트폴리오 웹페이지
  - `output/index.html`을 브라우저에서 직접 열어
    동작을 확인할 수 있어야 한다.
- **동작 조건**:
  - 별도의 서버 실행 없이 로컬 파일 시스템에서
    바로 동작해야 한다.
  - 파일을 넣으면 포트폴리오가 생성되고, 클릭하면
    바로 열리는 경험을 제공해야 한다.

## 거버넌스

- 이 Constitution은 프로젝트의 모든 설계·구현
  결정보다 우선한다.
- 원칙 위반이 필요한 경우, 반드시 사유를
  문서화하고 승인을 받아야 한다.
- Constitution 수정 시 다음을 준수한다:
  - 변경 사유와 영향 범위를 기록한다.
  - 시맨틱 버저닝을 적용한다 (MAJOR: 원칙 삭제·재정의,
    MINOR: 원칙 추가·확장, PATCH: 문구 수정·오탈자).
  - 관련 템플릿과 문서의 정합성을 검증한다.
- 모든 코드 리뷰에서 Constitution 준수 여부를
  확인해야 한다.
- 복잡성 추가는 반드시 정당화되어야 한다.
